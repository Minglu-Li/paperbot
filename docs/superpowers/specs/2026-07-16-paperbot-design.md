# Paperbot 研究资料库 MVP 设计文档

## 背景

Paperbot 是一个云端 SaaS 研究资料库，面向个人研究者的论文和研究资料管理场景。第一版重点不是做一个完整的研究代理平台，而是先把文献管理、跨设备阅读、标注、Markdown 笔记和受控 AI 工作流做扎实。

当前项目目录是一个绿地项目，目前仅包含设计文档，还没有应用代码。因此本设计按新项目规划。

## 产品目标

第一版要让用户可以：

1. 在云端上传和管理 PDF、Markdown、DOC/DOCX、TXT 等研究资料。
2. 用文件夹树和标签组织资料。
3. 跨设备访问资料库，桌面完整支持上传、阅读、标注、笔记和 AI，平板支持阅读标注，手机支持查找、阅读、笔记和 AI 问答。
4. 在线阅读 PDF 和统一文本视图资料，并进行高亮、颜色标记、Markdown 批注和摘录管理。
5. 为每个资料条目维护一篇主 Markdown 笔记，并允许把标注摘录插入主笔记。
6. 对扫描 PDF 进行 OCR，生成可搜索、可用于 AI 检索的文本。
7. 导出带标注的 PDF，或导出 Markdown 笔记和引用摘录。
8. 使用用户自己的模型 API key 调用 AI。
9. 让普通 AI 助手基于当前文件夹及子文件夹回答问题。
10. 让用户创建自己的 skill，并在运行 skill 时选择检索范围，默认范围是整个文件库。
11. 联网搜索论文网站，展示结果卡片和原始网页链接，但不绕过订阅或付费墙。

## 非目标

第一版不做以下内容：

1. 不做原生移动 App。
2. 不做团队协作 UI，但数据模型预留 workspace 和权限扩展。
3. 不做任意代码执行型 skill。
4. 不绕过出版社订阅、机构权限或付费墙下载论文。
5. 不做完整 Word 编辑器或专业 PDF 编辑器。
6. 不做复杂版面还原 OCR，例如多栏公式、复杂表格和图中文字的高精度结构化恢复。第一版 OCR 重点是让扫描 PDF 可搜索、可引用、可进入 AI 检索。
7. 不做公开 skill 市场。skill 是用户自定义、自己可见的能力单元。

## 产品范围

Paperbot 第一版采用“研究资料库 MVP”路线。

用户注册后进入个人 workspace。资料以文件夹树为主组织，每个文件属于一个主文件夹，同时可以拥有多个标签。用户上传文件后，系统保存原文件、抽取文本、建立全文索引和向量索引，然后支持阅读、标注、笔记和 AI 检索。

PDF 使用 PDF 阅读器展示；Markdown、DOC/DOCX、TXT 通过后台抽取为统一文本视图展示。非 PDF 文件第一版不追求原格式精细还原，而是优先支持一致的阅读、选中文本、高亮和 Markdown 批注体验。扫描 PDF 第一版纳入 OCR 流程：系统识别没有文本层或文本层质量很差的 PDF，后台生成 OCR 文本，并把 OCR 结果纳入全文搜索、RAG 和引用来源。

导出能力也是第一版范围。用户可以导出带标注的 PDF，也可以导出某个资料条目的 Markdown 主笔记、标注摘录和引用信息。导出不要求第一版支持复杂排版模板，但必须保证内容完整、来源可追溯，并且导出任务失败时可重试。

AI 由用户自己的模型 API key 驱动。默认 API key 仅保存在当前浏览器本地；用户可以主动开启跨设备加密同步。模型调用必须清楚显示当前 provider 和 model，避免用户不清楚成本来源。

普通 AI 对话默认绑定当前文件夹及子文件夹，严格基于该范围内的文件、标注、摘录和 Markdown 笔记回答。用户自定义 skill 是可复用工作流，运行时由用户选择检索范围，默认是整个文件库，也可以选择一个或多个文件夹、当前文件或当前选中文本。

## 架构设计

第一版建议采用一个 monorepo，里面包含多个可运行模块，而不是把所有逻辑塞进一个进程。

推荐结构：

```text
paperbot/
  apps/
    web/        # 前端 Web/PWA：页面、阅读器、笔记编辑器
    api/        # 后端 API：登录、文件夹、标注、笔记、AI 请求入口
    worker/     # 后台任务：PDF 解析、OCR、DOCX 转文本、生成索引、导出任务
  packages/
    db/         # 数据库 schema、迁移、查询封装
    shared/     # 前后端共用类型
    ai/         # RAG、模型 provider、skill 工具定义
```

“同一个工程”表示前端、API、worker 在同一个代码仓库中开发，方便共享类型和数据模型。“多个可运行模块”表示部署时至少有三个进程：`web`、`api`、`worker`。这样开发简单，但耗时任务不会阻塞用户请求。

核心模块包括：

1. **Auth / Workspace**
   负责用户登录、个人 workspace、未来团队 workspace 的成员和权限预留。

2. **Library**
   负责文件夹树、资料条目、标签、元数据、处理状态和列表筛选。

3. **Ingestion Pipeline**
   负责文件上传后的后台处理：保存原文件、抽取文本、扫描 PDF 的 OCR、生成阅读内容、切分检索片段、建立全文索引和向量索引。

4. **Reader / Annotation**
   负责 PDF 阅读器、统一文本阅读器、高亮、颜色、选区锚点、Markdown 批注、摘录和阅读位置同步。

5. **Notes**
   负责每个资料条目的主 Markdown 笔记，以及标注摘录插入主笔记。

6. **Export**
   负责导出带标注 PDF、Markdown 主笔记、标注摘录和引用信息。导出走后台任务，避免大文件或复杂标注导出阻塞用户请求。

7. **AI Gateway**
   负责统一接入用户自带模型 API key、模型 provider 配置、当前论文问答、选中文本解释、文件夹范围 RAG、标注生成笔记。

8. **RAG Scope Guard**
   负责强制所有 AI 检索先通过 workspace、用户权限和运行范围过滤，防止跨用户或跨权限访问。

9. **Paper Search**
   负责联网搜索论文网站并返回结果卡片和原始网页链接。不绕过订阅，不自动下载付费墙内容。

10. **User Skill Runner**
   负责执行用户自定义 skill。skill 只能调用平台白名单工具，不能执行任意代码。运行时用户选择范围，默认整个文件库。

## 数据模型

核心实体如下：

1. `User`
   用户账号。

2. `Workspace`
   工作区。第一版每个用户默认一个个人 workspace，未来可以扩展团队 workspace。

3. `Folder`
   树形目录，属于 workspace。用于资料组织和普通 AI 问答的默认范围选择。

4. `Document`
   资料条目，属于一个主文件夹，类型包括 PDF、Markdown、DOC/DOCX、TXT。记录标题、作者、年份、DOI、来源 URL、处理状态等元数据。

5. `FileObject`
   原始文件对象，记录存储位置、文件大小、MIME 类型、哈希、上传者和处理状态。

6. `DocumentText`
   抽取后的规范化文本。PDF 按页组织，文本类文档按块、标题层级或段落组织。

7. `Annotation`
   标注记录，包括高亮颜色、选中文本、页码、文本 offset、块 id、PDF 锚点或统一文本锚点。

8. `AnnotationComment`
   标注上的 Markdown 批注。

9. `DocumentNote`
   每个资料条目的一篇主 Markdown 笔记。

10. `Tag` / `DocumentTag`
    标签和资料条目之间的多对多关系。

11. `SearchChunk`
    全文搜索和 RAG 的最小片段，关联来源 document、页码/位置、标注或笔记来源。

12. `EmbeddingIndex`
    向量索引记录，必须包含 workspace、document、folder path 或 folder id 信息，便于权限和范围过滤。

13. `AIProviderConfig`
    用户选择的 provider、model 和模型参数。API key 默认保存在浏览器本地；如果用户开启跨设备同步，则保存服务端加密版本。

14. `UserSkill`
    用户自定义 skill，包含名称、说明、输入定义、提示词/步骤、允许调用的工具、版本和拥有者。

15. `SkillRun`
    每次 skill 运行记录，包含运行用户、skill 版本、运行时范围、输入、工具调用日志、输出和错误信息。

16. `OcrJob`
    扫描 PDF 的 OCR 任务记录，包含 document、文件版本、语言配置、状态、错误信息和 OCR 文本质量指标。

17. `ExportJob`
    导出任务记录，包含导出类型、范围、状态、生成文件位置、错误信息和请求用户。

## 关键数据流

### 文件导入

1. 用户上传 PDF、Markdown、DOC/DOCX 或 TXT。
2. API 保存原文件到对象存储，创建 `Document` 和 `FileObject`，状态设为 `processing`。
3. API 将解析任务放入后台队列。
4. `worker` 抽取文本，生成 `DocumentText`。
5. 如果 PDF 没有可用文本层或文本质量过低，`worker` 创建 `OcrJob` 并执行 OCR。
6. OCR 成功后，系统把 OCR 文本写入 `DocumentText`，并记录文本来源为 OCR。
7. `worker` 切分 `SearchChunk`，建立全文索引和向量索引。
8. 处理成功后，`Document` 状态变为 `ready`。
9. 处理失败时，状态变为 `failed` 或 `needs_attention`，保留原文件，显示失败原因并允许重试。

### 阅读与标注

1. 用户打开资料。
2. PDF 走 PDF 阅读器；Markdown、DOC/DOCX、TXT 走统一文本阅读器。
3. 用户选择文本并创建高亮，系统保存 `Annotation`。
4. 用户为标注添加 Markdown 批注，系统保存 `AnnotationComment`。
5. 用户编辑主笔记，系统保存 `DocumentNote`。
6. 用户可以把某条标注摘录插入主笔记，插入内容保留来源文件、页码或文本位置。

### 导出

1. 用户选择导出当前资料、当前文件夹下的笔记，或某个资料条目的标注摘录。
2. 导出带标注 PDF 时，系统读取原 PDF、标注位置、颜色和批注引用，生成新的 PDF 文件。
3. 导出 Markdown 时，系统组合主笔记、标注摘录、批注和引用来源，生成 Markdown 文件。
4. API 创建 `ExportJob`，由 `worker` 在后台生成导出文件。
5. 导出完成后，前端展示下载入口；导出失败时展示失败原因并允许重试。

### 普通 AI 问答

1. 用户在某个文件夹下向 AI 提问。
2. 后端将检索范围固定为该文件夹及其子文件夹。
3. `RAG Scope Guard` 先按 workspace、用户权限和 folder scope 过滤可检索内容。
4. RAG 检索 `SearchChunk`，包括文件正文、主笔记、标注和批注。
5. `AI Gateway` 调用用户选择的模型。
6. 回答必须附带引用来源。若当前范围没有足够证据，必须明确说明无法基于当前资料回答。

### 用户 skill 运行

1. 用户创建自己的 skill，定义名称、说明、输入项、提示词/步骤和可调用工具。
2. 用户运行 skill 时选择范围，默认是整个文件库，也可以选择一个或多个文件夹、当前文件或当前选中文本。
3. `User Skill Runner` 解析运行范围并执行权限检查。
4. skill 只能调用平台白名单工具，如资料检索、当前论文问答、总结文件、解释选中文本、从标注生成笔记、联网搜索论文等。
5. 每次工具调用都记录到 `SkillRun`。
6. skill 输出必须保留可追溯来源。

### 联网论文搜索

1. 用户或 skill 发起论文搜索。
2. 系统搜索合法公开网页和论文索引。
3. 前端展示结果卡片，包括标题、作者、年份、摘要片段、来源网站和链接。
4. 用户点击卡片跳转原网页。
5. 系统不绕过订阅，不自动下载付费墙内容。
6. 用户如果拥有合法 PDF，可以手动上传，之后再进入阅读、标注和 AI 流程。

## AI 与 BYOK 策略

AI 能力必须围绕用户自己的 API key 运行。第一版支持用户在设置页选择 provider、model，并粘贴 API key。

API key 策略：

1. 默认只保存在当前浏览器本地。
2. 用户可以主动开启跨设备加密同步。
3. 开启同步后，服务端只保存加密后的 key。
4. UI 必须清楚展示当前使用的 provider/model。
5. 所有模型调用都消耗用户自己的 key。

普通 AI 与用户 skill 的区别：

1. 普通 AI 是即时问答，默认跟随当前文件夹上下文。
2. 用户 skill 是可复用工作流，运行时选择范围，默认整个文件库。
3. skill 能调用多个平台工具，但不能执行任意代码。

## 安全边界

1. 所有资料、文件、标注、笔记、索引、skill 都必须绑定 `workspace_id` 和拥有者或权限信息。
2. 所有 API 请求都必须做 workspace 和用户权限检查。
3. 全文索引和向量索引不能绕过数据库权限过滤。
4. 普通 AI 只能访问当前文件夹及子文件夹。
5. 用户 skill 默认可以检索整个文件库，但仍然只限用户有权限访问的资料。
6. skill 只能调用平台白名单工具。
7. 第一版不允许用户上传 JS/Python 等任意代码执行。
8. 联网论文搜索不绕过订阅、机构权限或付费墙。
9. 回答和 skill 输出必须能追溯来源，避免不可验证的结论。
10. API key 默认只保存在本地；跨设备同步必须由用户主动开启。

## 错误处理

长任务状态需要清楚呈现：

1. `processing`：文件正在解析或建索引。
2. `ready`：文件已准备好，可阅读、标注、搜索和用于 AI。
3. `failed`：处理失败，显示失败原因，允许用户重试。
4. `needs_attention`：文件可保存但需要用户处理，例如 OCR 失败、文本质量过低或文件格式异常。

具体策略：

1. 文件解析失败时保留原文件。
2. 后台任务可重试，并记录失败原因。
3. OCR 失败时保留原文件和已有解析结果，显示失败原因，并允许用户重试 OCR。
4. 导出失败时保留导出任务记录，显示失败原因，并允许用户重新导出。
5. AI 检索不到证据时不得编造，必须说明当前范围内没有足够依据。
6. 模型调用失败时展示 provider、model 和错误类型，但不泄露 API key。
7. skill 调用失败时记录失败步骤和工具调用日志，方便用户调试。
8. 标注锚点失效时尽量通过文本匹配恢复；无法恢复时显示“位置需要重新确认”。

## 测试策略

第一版至少覆盖以下测试：

1. **权限测试**
   验证用户不能访问其他用户 workspace、文件、标注、笔记、索引或 skill。

2. **上传解析测试**
   覆盖 PDF、Markdown、DOCX、TXT 的上传、抽取文本、状态变更和失败重试。

3. **OCR 测试**
   覆盖扫描 PDF 检测、OCR 任务创建、OCR 文本写入、索引生成、失败重试和 OCR 质量提示。

4. **阅读与标注测试**
   验证 PDF 高亮、文本视图高亮、颜色、Markdown 批注、主笔记、摘录插入和阅读位置同步。

5. **标注锚点测试**
   验证刷新、缩放、换设备后标注仍能定位；失效时有降级提示。

6. **导出测试**
   验证带标注 PDF 导出、Markdown 笔记导出、引用来源保留、导出任务失败重试和无权限导出拦截。

7. **RAG 范围测试**
   验证普通 AI 只检索当前文件夹及子文件夹，不能越过范围。

8. **用户 skill 测试**
   验证 skill 默认整个文件库、可选择多个文件夹、只能调用白名单工具、不能跨权限访问。

9. **BYOK 测试**
   验证本地保存、跨设备加密同步、provider/model 选择、错误处理和 key 不泄露。

10. **端到端测试**
   覆盖主流程：上传文件 -> 后台处理/OCR -> 阅读标注 -> 写 Markdown 笔记 -> 文件夹 AI 问答 -> 运行用户 skill -> 导出带标注 PDF 或 Markdown 笔记。

## 后续扩展方向

1. 团队 workspace、共享资料库、协作标注和 workspace 可见 skill。
2. BibTeX/RIS/CSL JSON 批量导入。
3. 更高精度的 OCR 版面结构化、公式识别和表格恢复。
4. 可配置导出模板、批量导出和引用格式样式。
5. 更强的移动端阅读体验或原生 App。
6. 更完整的外部论文数据库集成。
7. 受控代码沙箱型 skill，但这需要单独设计安全模型。
